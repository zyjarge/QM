# 00 · 产品概述与定位

> 文档版本：v1.0 | 日期：2026-07-17 | 状态：基线草稿

---

## 1. 产品名称

**QM（Requirement Manager / 需求管家）**
工作代号：QM。定位语：**群即现场，平台即档案；审批在指尖，留底全自动。**

---

## 2. 要解决的三个核心痛点

| # | 痛点 | 根因 | 现状后果 |
|---|------|------|----------|
| P1 | 业务方说不清需求 | 缺产品思维，描述停留在"我要XX功能"，缺场景/规则/边界/验收标准 | 研发自行脑补，理解偏差 |
| P2 | 研发过程做偏返工 | 无书面基线、无确认关卡，口头/聊天传递 | 偏差在交付期才暴露，数倍返工成本 |
| P3 | 需求资产查不到 | 需求散落在碎片化聊天会话，无归档、无版本、无索引 | 无法回溯决策依据，知识不沉淀 |

## 3. 产品定位（差异化核心）

QM **不是**又一个 Jira/PingCode 克隆。它是**以需求为轴心的会话式协作平台**：

```
传统模式（通知驱动）：
  事情发生在平台 → 消息推送到 IM → 人回平台操作

QM 模式（会话驱动）：
  需求生成 → 自动建群（一需求一群）
         → 沟通/澄清/评审/签认全部在群内完成
         → 平台是"隐形书记员"：旁听群内一切，自动留底、驱动状态机
         → 需求关闭 → 群自动解散，完整记录归档可查
```

**五条设计铁律：**

1. **SSOT（唯一事实源）**：需求的权威版本在平台，不在任何聊天记录里
2. **引导式采集**：不给业务方空白文档，用模板 + AI 对话引导"把话说全"
3. **无基线不排期**：未经业务负责人基线签认的需求，不允许进入开发
4. **全链路可追溯**：需求 ↔ 群聊记录 ↔ 任务 ↔ 代码 ↔ 测试 ↔ 发布
5. **IM 是入口，平台是底座**：人在飞书/企微里完成全部流程，数据沉淀在平台

## 4. 目标用户与角色

| 角色 | 职责 | 主要触点 |
|------|------|----------|
| 业务提出人（Requester） | 提出需求、参与澄清、验收 | IM 群 / 机器人对话 |
| 业务负责人（Biz Owner） | 需求评审、**基线签认**、变更审批 | IM 卡片（一键操作） |
| 产品经理（PM） | 需求整理、文档撰写、流程推进 | Web 端 + IM 群 |
| 研发负责人（Dev Lead） | 技术评审、任务拆解、排期 | IM 群 + Web 端 |
| 测试负责人（QA） | 验收标准评审、测试用例关联 | IM 群 + Web 端 |
| 管理员（Admin） | 模板配置、权限管理、报表 | Web 端 |

## 6. 成功度量（上线后验证）

| 指标 | 定义 | 目标 |
|------|------|------|
| 返工率 | 基线后变更次数 / 需求总数 | 较现状下降 50% |
| 一次评审通过率 | 首轮评审全票通过的需求占比 | ≥ 60% |
| 需求澄清周期 | 提出 → 已基线的平均时长 | ≤ 3 个工作日 |
| 需求要素完整率 | 提交时通过模板校验的需求占比 | ≥ 90% |
| 回溯响应时长 | "查一个历史需求决策依据"的耗时 | ≤ 1 分钟 |

## 7. 范围边界（明确不做什么）

- **不做**项目管理全套（不做甘特图/资源管理/工时统计——聚焦需求生命周期）
- **不做** IM 本身的替代品（不建独立聊天功能，寄生在飞书/企微之上）
- **P0 不做**多租户、开放平台、钉钉（见 07-roadmap）
- **不替代**代码托管/CI（只做事件对接与追溯关联）

## 8. 名词约定

| 术语 | 定义 |
|------|------|
| 需求（Requirement） | 平台管理的核心实体，有唯一编号 REQ-YYYY-NNNN |
| 基线（Baseline） | 业务负责人签认的需求内容快照，签后锁定 |
| 内容指纹 | 基线时需求正文的 SHA-256 摘要，签认凭证 |
| 需求群 | 为单个需求自动创建的 IM 群聊 |
| 书记员引擎 | 监听群消息事件、自动归档/打标/驱动状态机的服务模块 |
| 变更单（Change Request） | 基线后对需求的修改申请，含 diff 与影响评估 |
