# 08 · 产品应用组件与输出物

> 文档版本：v1.0 | 日期：2026-07-17

## 1. 产品应用层组件

QM 最终交付给用户的是 **5 个应用组件**：

```
                   ┌─────────────────┐
                   │   飞书 H5 微应用  │ ← 移动端轻量查看
                   └────────┬────────┘
                            │
  ┌───────────┐    ┌────────┴────────┐    ┌───────────┐
  │ 飞书机器人   │◄──│    后端服务     │──►│ OpenAPI   │
  │ (群内交互)  │    │  (Spring Boot) │   │ (系统对接) │
  └───────────┘    └────────┬────────┘   └───────────┘
                            │
                   ┌────────┴────────┐
                   │  Web 管理后台    │ ← 主力操作界面
                   │  (React SPA)    │
                   └─────────────────┘
```

### 1.1 Web 管理后台（React SPA）

**面向角色：** PM / 管理者 / 需要深度操作的人员

| 模块 | 页面/功能 | 说明 |
|------|----------|------|
| 登录 | 飞书免登 / 账号密码登录 | 首次登录后自动跳转 |
| 需求池 | 列表视图 + 看板视图 | 按状态/类型/产品线/时间/干系人筛选；批量操作 |
| 需求详情 | TipTap 块编辑器 | 富文本+表格+流程图+附件；版本树 + diff 对比 |
| 验收标准 | Given/When/Then 或 Checklist 管理 | 每个版本独立关联 |
| 干系人管理 | 增删改干系人（联动 IM 群成员） | 自动拉群/踢群 |
| 时间线 | 生命周期事件流 | 从提需求到归档的完整记录 |
| 群消息归档 | 按时间线展现的只读群聊记录 | 支持全文搜索 |
| 追溯矩阵 | 需求 ↔ 任务 ↔ commit ↔ 测试 ↔ 发布 | 关联链路可视化 |
| 变更单 | 基线后修改记录 + diff | 变更历史可查 |
| 待办中心 | 待评审/待签认/待验收/待审批 | 超时标记 + 一键跳转 |
| 需求库搜索 | 全文搜索 + 语义搜索（P2） | 贯穿需求+消息+评审意见 |
| 模板管理 | 需求类型/动态字段配置 | 管理员配置 |
| 报表看板 | 吞吐量/返工率/一次通过率/周期/质量分 | 管理视角度量 |
| 系统设置 | IM 配置/Webhook/备份管理 | 管理员 |

### 1.2 飞书 H5 微应用

**入口：** 飞书工作台 → 点 QM 图标

**面向角色：** 全员（尤其是业务提需人、业务负责人）

**特点：**
- 飞书免登，打开即用
- Web 管理后台的轻量子集，不做全部复刻
- 核心页面：
  - 我的需求列表（只展示与我相关的需求）
  - 需求详情（只读 + 评论区）
  - 我的待办
  - 从 H5 发起提需求（跳转到机器人对话或提交表单）

### 1.3 飞书机器人（@QM 助手）

**入口：** 需求群内 @QM 助手 或 与机器人私聊

**面向角色：** 全员 —— 日常 80% 的操作在此完成

| 能力 | 说明 |
|------|------|
| 对话引导提需求 | 多轮问答补齐要素，自动生成需求草稿 |
| 群内交互卡片 | 评审/签认/变更审批/超时提醒（按钮一键操作） |
| 状态播报 | 需求流转时自动群内通知 |
| AI 书记员 | 识别群内决策性发言，提示"是否录入需求文档" |
| 解散预警 | 需求归档前 24h 群公告提醒 + 归档链接 |
| 自然语言操作 | "@助手 总结本周需求进展" / "整理刚才的讨论为验收标准" |

### 1.4 OpenAPI + Webhook

**面向：** 外部系统（CI / Git / OA / 第三方集成）

| 接口 | 说明 |
|------|------|
| RESTful API 全集 | 需求 CRUD / 状态流转 / 流程操作 / 查询搜索 |
| 事件 Webhook | 需求状态变更 / 评审通过 / 基线签认 / 变更审批 |
| 认证方式 | H5 端 JWT；系统对接 HMAC 签名 |
| 文档 | OpenAPI 3.0（Swagger） |

### 1.5 后端服务集群

用户不可见，但构成应用层的基础骨架：

| 服务 | 职责 | 技术 |
|------|------|------|
| qm-server | 全部业务逻辑：需求/流程/群引擎/书记员/AI/追溯/报表 | Java 17 + Spring Boot 3 |
| qm-job | MQ 异步消费者：消息归档/通知发送/超时检查/群解散定时器 | 同 qm-server 镜像 |

## 2. 产品输出物清单

QM 作为一个需求全生命周期管理平台，最终产出的"东西"按资产类型分为六类：

### 2.1 IM 端输出物（用户日常交互触点）

| 输出物 | 触发时机 | 用户感知 |
|--------|---------|---------|
| **需求群** | 需求进入"待澄清"状态 | 自动建群拉人，群公告实时更新状态 |
| **评审卡片** | 发起评审时 | 群内卡片，含通过/有意见/转交按钮，进度实时更新 |
| **基线签认卡片** | 评审通过后 | 含内容指纹 SHA-256，一键确认或退回 |
| **变更审批卡片** | 基线后申请修改 | minor 快速通道 / major 重回评审 |
| **状态播报卡片** | 每次状态流转 | 群内通知：谁做了什么、到了什么状态 |
| **超时升级卡片** | 48h/72h 未处理 | 自动提醒本人 → 升级抄送上级 |
| **解散预警卡片** | 需求归档前 24h | "本群将于 24h 后解散，归档链接：..." |
| **AI 对话回复** | @QM 助手 时 | 多轮引导、自动整理、提示录入 |

### 2.2 Web 端输出物（管理界面）

| 输出物 | 说明 |
|--------|------|
| **需求池** | 结构化列表 + 看板视图，全部需求一览 |
| **需求详情页** | 版本化文档、验收标准、干系人、时间线、群归档、追溯、变更 |
| **版本树 + diff** | 任意两版逐行对比 |
| **我的待办** | 个人驱动的审批/评审/验收任务聚合 |
| **需求库搜索结果** | 全文/语义检索命中内容 |
| **报表** | 吞吐量、返工率、一次通过率、周期、积压 |
| **归档包查看页** | 已完成需求的完整档案（只读） |

### 2.3 归档与证据输出物（数据资产沉淀）

| 输出物 | 格式 | 说明 |
|--------|------|------|
| **需求归档包** | 结构化数据（DB）+ MinIO | 需求文档终版 + 版本历史 + 群消息全量记录 + 附件 |
| **基线签认记录** | DB 永久行 + 内容指纹 | SHA-256 签认快照，签字人/时间/设备信息 |
| **审计日志** | DB 只增不改 | 全部写操作 + IM 卡片交互，不可篡改 |
| **基线 PDF 导出（P1）** | PDF 水印 | 文字签认存证，可作法律依据 |

### 2.4 管理报表输出物（度量与复盘）

| 指标 | 定义 | 价值 |
|------|------|------|
| **需求吞吐量** | 月度/季度完成需求数 | 团队产能趋势 |
| **返工率** | 基线后变更次数 / 需求总数 | **核心痛点量化**——第一次说清返工有多严重 |
| **平均需求周期** | 提出→基线→交付各阶段耗时 | 瓶颈在哪一段（澄清/评审/开发/验收） |
| **一次评审通过率** | 首轮全票通过的需求占比 | 需求质量"体温计" |
| **需求质量分** | 一次提报要素完整度 + 澄清轮次 | 业务方需求描述能力的量化反馈 |
| **验收积压** | 待验收超期需求列表 | 哪一步推不动了 |

### 2.5 集成输出物（开放接口）

| 输出物 | 说明 |
|--------|------|
| **OpenAPI 文档** | Swagger 界面，供集成方查阅调用 |
| **Webhook 事件推送** | 状态变更/评审/基线等事件实时投递 |

### 2.6 平台内部数据资产

| 资产 | 说明 |
|------|------|
| **结构化需求库** | 每一个需求的完整数字化档案，不再是碎片化聊天 |
| **组织通讯录映射** | 多 IM 身份关联的企业组织信息 |
| **追溯矩阵** | 需求 ⇄ 代码 ⇄ 测试 ⇄ 发布的完整关联图 |

## 3. 不同角色看到的"产品"是不同的

| 角色 | 他们看到的产品 |
|------|---------------|
| 业务提需人 | 飞书机器人（群内@） + H5 看自己提的需求 |
| 业务负责人 | 飞书卡片点"签认/审批" + 偶尔打开 Web 看报表 |
| PM | Web 管理后台 + 飞书群驱动流程 |
| 研发/测试 | 飞书群内收卡片 + Web 做评审/关联任务 |
| 管理员 | Web 管理后台全功能 + 系统设置 |
| 外部系统 | OpenAPI + Webhook |

## 4. 关键设计原则

- **80/20 原则**：日常 80% 的操作（评审/签认/审批/通知）在 IM 群内通过卡片完成，不打开 Web
- **IM 是入口，平台是底座**：人在飞书里交互，数据自动沉淀在平台
- **一站式归档**：一个需求关闭时，生成一份完整档案（文档+消息+审批+签认+变更），永不丢失
